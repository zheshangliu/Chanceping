# 安全策略

感谢你帮助保持 ChancePing 项目的安全。本文档描述了如何报告安全漏洞以及我们处理漏洞的流程。

---

## 1. 报告漏洞

**请勿通过公开的 GitHub issue 报告安全漏洞。**

如果你发现安全漏洞，请通过以下方式私下报告：

### 方式一：GitHub Security Advisory（推荐）

1. 前往本仓库的 Security 标签页
2. 点击 "Report a vulnerability"
3. 填写漏洞详情

### 方式二：邮件

发送邮件至项目维护者（见仓库 profile），主题加上 `[SECURITY]` 前缀。

### 报告内容

请尽可能包含以下信息：

- 漏洞类型（如 XSS、SQL 注入、信息泄露、权限绕过等）
- 受影响的版本/文件/端点
- 复现步骤（最好附带 PoC）
- 影响范围（可被攻击者利用做什么）
- 建议的修复方案（可选）

---

## 2. 响应流程

1. **确认收到**：维护者会在 48 小时内确认收到报告
2. **初步评估**：3 个工作日内给出初步评估（是否接受、严重程度）
3. **修复开发**：接受后，维护者开发修复补丁，期间与报告者保持沟通
4. **发布修复**：补丁测试通过后发布新版本
5. **公开披露**：修复发布 90 天后（或与报告者协商的更早时间），公开披露漏洞详情

---

## 3. 严重程度分级

参考 [CVSS v3.1](https://www.first.org/cvss/) 评分标准：

| 等级 | CVSS 评分 | 说明 |
|---|---|---|
| 严重（Critical） | 9.0-10.0 | 远程代码执行、敏感数据大规模泄露 |
| 高危（High） | 7.0-8.9 | 权限绕过、SQL 注入、重要数据泄露 |
| 中危（Medium） | 4.0-6.9 | XSS、CSRF、有限的信息泄露 |
| 低危（Low） | 0.1-3.9 | 信息泄露（非敏感）、配置问题 |

## 4. 支持的版本

本项目处于 V1.0 阶段，仅对最新发布版本提供安全更新。

| 版本 | 安全支持 |
|---|---|
| 1.0.x | ✅ 支持 |
| < 1.0 | ❌ 不支持 |

## 5. 安全最佳实践（部署者）

部署 ChancePing 时请遵循以下安全实践：

### 环境变量

- `.env` 文件权限设为 600（仅所有者可读写）
- 不要将 `.env` 提交到 Git（已在 `.gitignore` 中忽略）
- API Key 定期轮换
- 生产环境使用独立的 API Key，不与开发环境共用

### 网络配置

- 不要将开发端口（3000）直接暴露到公网
- 使用反向代理（Nginx / Caddy）转发请求
- 启用 HTTPS（TLS 1.2+）
- 配置 CORS 白名单

### 数据保护

- `data/` 和 `reports/` 目录权限设为 700
- 定期备份 `data/` 目录
- 日志中不要打印 API Key 或用户敏感信息

### 依赖更新

- 定期运行 `npm audit` 检查依赖漏洞
- 关注 [GitHub Dependabot](https://docs.github.com/en/code-security/dependabot) 告警
- 及时更新有漏洞的依赖

## 6. 已知安全设计

### Mock 模式

- 默认 `NOTIFY_MOCK_MODE=true`，所有通知仅打印日志不实际发送
- 无 API Key 时所有 LLM / 搜索 Provider 自动降级为 Mock 模式
- 参赛版默认 `SCHEDULER_ENABLED=false`，避免自动触发外部请求

### 输入校验

- API 路由使用 JSON Schema 校验请求体
- URL 输入经过 `src/utils/link-validator.ts` 校验
- DSL 解析器对恶意语法有防护

### 输出安全

- 报告导出 HTML 时转义特殊字符，防 XSS
- 日志脱敏：API Key 在日志中以 `***` 显示
